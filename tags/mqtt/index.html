2. 消息格式

所有mqtt消息都有固定头部（fix header）。一部分mqtt消息包含可变头部（variable header）和消息负载（payload）。下面的部分组要介绍mqtt消息的组成部分。

2.1 固定头部（fixed header）

所有mqtt消息都包含有fixed header，fixed header定义如下：

  1. byte1 包含Message Type和Flags（DUP QOS RETAIN）。

  2. byte2 （至少包含一个byte）包括剩余字段的长度。所有的数据类型是以大端对齐(big-endian)高位字段排在低位字段的前面。例如一个16bit数字，在网络上以最高有效字节（MSB）在前，最低有效字节(LSB)在后的方式传输。

    Dup： 当客户端或者server尝试重新发送PUBLISH/PUBREL/SUBSCRIBE/UNSUBSCRIBE消息时，并且消息的QoS大于0，需要将DUP置位，表示这是一个重复发送的消息。当DUP被置位时，可变头部会包含一个MessageId字段。

    接收端使用DUP确定消息是否可能已经被接收过的一个提示。但是这个flag不能用作检测消息是否真的被接收过。

    RETAIN：这个flag用于PUBLISH消息。当客户端向服务端发送一个Retain被置位的消息，服务端接收消息并将消息下发在线接收者后，还需要保留消息。

    当新客户端订阅topic时，当前topic下最后一条Retained消息将会下发给客户端，下发Retain消息时Retain flag置位；如果没有Retianed消息，就无需下发。

    Retain消息可以用于report by exception，或者，希望新的订阅者能够立即接到最新的消息时使用。

    当Retain消息发送给在线的订阅者时，Retain flag应当不置位，这有利于客户端区分这条消息是最新的（live的）而不是先前产生的。

    在服务端从新启动后，要保证retained消息依然有效。

    如果服务接到一个payload长度为0的Retained消息，server即可删除先前的Retain消息（即清空Retain消息）。

    2.1.3 Remaining length

    剩余消息的长度包括可变消息头部和消息负载。

    Remaining length由1-4个bytes组成，具体长度与表示的数值有关。其中每个byte的第一位是标识位，如果是0表示当前byte为Remaining length的最后一个byte，如果是0表示，当前byte后面还有额外的bytes表示长度。因此，如果是一个byte最大能表示0x7F（0111,1111）即127，两个byte能够表示0xFF 0x7F（1111 1111 0111 1111）转换成实际的二进制位( 111 1111 111 1111 -> 11 1111 1111 1111 -> 0x3F 0xFF）即为16383，因此能够表示的最大数字为0xFF 0xFF 0xFF 0x7F(1111 1111 1111 1111 1111 1111 0111 1111 -> 1111 1111 1111 1111 1111 1111 1111->x0F 0xFF 0xFF 0xFF)即为268435455。如下表所示：

2.2 variable header

一些类型的mqtt消息包含variable header，它位于fix header和payload之间。

2.2.1 Protocol name

protocol name存在于MQTT CONNECT消息的variable header中。这个字段使用UTF-encoded String，内容为MQIsdp。

2.2.2 Protocol Version

Protocol version 存在于MQTT CONNECT消息的variable header中，使用8-bit无符号整数，表示当前客户端使用的mqtt协议版本，当前mqtt版本为3(0x03).

2.2.3 Connect flags

Connect flags 由clean session，will，will Qos和Retainflags表示，存在于connect消息的variable header中。
clean session flag：如果设置为0，当客户端离线时，服务器需要存储其订阅的QoS1和QoS2消息，当客户端再次连接时，可以接收这些离线消息，同时service需要处理在客户端断开连接时正在发送的消息（in-flight），这些消息需要一直保留到客户端下次重新连接。

如果clean session flag置位，服务器在客户端连接时必须抛弃所有先前和客户端相关的信息，当客户端断开连接后，服务器要清除任何和客户端相关的信息。

通常，一个客户端只会使用一种模式，并且与具体的应用场景有关。clean session client不会接受过期的消息，并且每次连接的时候需要从新订阅相应的topic。non-clean session client将不会丢失任何QoS1和QoS2消息，即使是在他离线情况时发送的消息。

这个flag原名为“clean start”，现在重新命名，以区分他的作用范围为整个连接过程，而不仅仅是初始链接过程。

服务器可以有一些策略用户清理那些可能永远都不会重新连接的客户端相关的信息。



Will flag：will flag置位，当服务端与客户端交互过程中遇到ioerror，或者客户端在keep alive timer时间内失去连接，服务端会以客户端的名义往指定的topic下面发送一个消息。如果客户端发送disconnect消息断开连接，则不会触发will消息。

如果will flag 置位，will QoS，will Retain在connect flags中必须出现，并且will topic和will message fields必须出现在payload中。

will QoS：只有will flag被置位，will QoS才有意义，will QoS可以取值0,1,2.表示异常情况下，服务端代发的will消息的QoS

will Retain flag:与will QoS相似，仅有will flag被置位时，才有意义，表示异常情况下，服务端代发的will消息的QoS

User name和Password flags：客户端链接过程可以指定user name和password，将user name和password置位，那面user name和password就会下connect的payload中。

如果user name置位，那么user name必须在payload中，同理，如果password置位，password必须在payload中，仅设置password不设置user name是无效的。

2.2.4 keep Alive timer

Keep Alive timer存在于Mqtt connect的可变头部中。

Keep Alive timer以分钟为单位，定义从客户端接收消息最大的时间间隔。保证服务器能够有效的监测客户端已经断开连接，而不是等待很长时间的TCP/IP超时。

客户端需要保证在Keep Alive time周期内向服务端发送消息，即使客户端没有需要向服务端提交的数据，也要发送PINGREQ消息，服务端回馈客户端PINGRESP消息

如果服务端在1.5倍的Keep alive time周期内没有接收到客户端的任何消息，服务端断开与客户端的链接。

同理，如果客户端发送PINGREQ后，在Keep Alive time周期内没有收到服务端的PINGRESP，客户端盈动断开socket链接。

Keep Alive time由16bit的值表示，以分钟为单位。根据应用场景的不同，设置不同的值，通常设置为几分钟，最大时间可以设置为18小时，设置为0，表示客户端不会断开链接。

2.2.5 connect return code

connect return code 存在于CONNACK消息的variable header中。定义为一个byte无符号数值,0表示成功。

2.2.6 topic name

topic name存在于PUBLISH消息的可变头部中。

topic name定义了payload发往的通道信息。订阅者使用topic name说明要接受的信息通道。topic name是一个utf-8 string。最长可为32767.

2.3 Payload

下面的消息具有payload。

2.3.1 connect

payload包含一个或多个UTF-8string，包括client unqiue id，will topic，will message，user name，password。除了client unqiue id外，其他都是可选的，这与variable header中的flag有关。

2.3.2 subscribe

包含客户端希望订阅的topic的名字和QoS级别，使用UTF-encoded。

2.3.3 suback

包含一个QoS级别列表，服务端保证客户端可以接受topic的一个Qos级别列表，并且次序与subscribe一致。

2.3.4 publish

publish payload仅包含应用程序相关的数据，并且这部分数据被作为一个Blob。如果应用程序需要理解payload中的数据，应用程序需要单独定义payload字段的解析规则，不能通过修改fixheader或者variableheader中的字段。

2.4 message identifiers

message identifier会攒在与PUBLISH/PUBACK/PUBREC/PUBREL/PUBCOM/SUBSCRIBE/SUBACK/UNSUBSCRIBE/UNSUBACK中。
